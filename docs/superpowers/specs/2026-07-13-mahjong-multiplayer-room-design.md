# 麻将计分多人房间改造设计

## 背景

现有 iOS 客户端已经具备创建房间、扫码加入、WebSocket 同步、给分、台板、转让房主和结算界面的雏形，但线上 Cloudflare Durable Object 与客户端消息协议不一致：客户端发送的 `give_score`、`add_player`、`remove_player`、`table_score`、`transfer_owner` 尚未被线上 Worker 完整处理。结算接口也错误地先调用只会结束房间的 `/end`，导致倍率、胜负和倍率分没有被写入。

本次改造参考用户提供的小程序录像，保留 PartyGames 现有视觉语言，用 iOS 26 原生 Liquid Glass 完成多人房间的完整闭环，并修复线上协议、权限和异常状态。

## 目标

- 房主创建多人房间后，初始列表只显示房主本人。
- 其他玩家必须通过二维码或网页邀请链接加入，加入成功后实时出现在所有设备上。
- 房间内支持原子给分、台板、玩家退出或踢出、转让群主、语音播报、明细和倍率结算。
- 房主在房间有效期内可随时重新打开同一个有效二维码，不因关闭邀请弹层而失效。
- 网络异常、被移出、房间结束和服务不可达必须显示不同状态，不能把普通断网误报为“你已退出房间”。
- iOS 客户端和线上 Cloudflare Worker 使用同一套协议与权限规则。

## 非目标

- 不复制参考小程序的紫色主题、广告或品牌元素。
- 不重做应用首页、游戏库或其他非房间模块。
- 不引入新的本地 Node 服务作为生产依赖；`server/mahjong-sync-server` 只保留为行为参考或本地测试工具。
- 不在本次改造中增加聊天、好友、观战或跨房间排行榜。

## 已确认的交互

### 房间页

- 顶部显示房间标题、房间号和实时连接状态。
- 工具区依次提供“添加玩家”“转让群主”“台板”“语音播报”。
- “添加玩家”不创建虚拟玩家，点击后从底部弹出邀请 sheet。
- “转让群主”仅房主可用，候选人必须是已扫码加入、仍处于房间内的真实成员。
- “台板”和“语音播报”使用一致的 iOS 26 Liquid Glass 控件。
- 多人房间创建完成时只有房主一行；扫码加入后才增加玩家行。
- 玩家行中的“给分”打开居中的紧凑小弹框，不进入新页面，也不使用大尺寸 sheet。
- 底部保留“给分详情”和“结算房间”。

### 邀请弹层

- 邀请视图使用 `.sheet` 从底部弹出，保持 iOS 的上下进出动效，不进入独立导航页面。
- 二维码编码的内容与下方网页链接完全一致，均为 HTTPS 加入地址。
- 原“邀请码”位置改为完整网页链接；点击整行立即复制，并给出复制成功反馈。
- 二维码和链接在房间处于 `active` 状态时持续有效。房主关闭后可随时通过“添加玩家”重新打开。
- 房间结算后，旧链接只展示房间已结束，不允许创建新成员。

### 给分

- 弹框包含目标玩家、整数输入框、双方变更预览、取消和确认。
- “随机数字”按“允许用户输入任意正整数”实现，不固定为 20；界面示例可以显示 13、27 等不同数值。
- 有效范围为 `1...1_000_000`。零、负数、小数、空值和越界值不能提交。
- 确认后从操作者自己的玩家账户扣除同额分数，并给目标玩家增加同额分数；总分保持为零。
- 提交期间禁用确认按钮。客户端不自动重发未确认操作，重连后以服务端快照为准，避免重复给分。

### 台板与语音播报

- 台板开关是每台设备的操作入口偏好；开启后显示给台板的操作。第一次给台板分时，服务端自动创建房间共享的“台板”账户。
- 台板给分仍是零和转账：操作者扣分，台板增加同额分数，并产生双方流水。
- 语音播报是每台设备的本地偏好，不写入房间状态。
- 只播报连接期间新收到的给分事件，不重播重连后拉取的历史流水。
- 播报文案包含来源、目标和分数，例如“阿龙给小雨 13 分”。

### Liquid Glass 与反馈

- 项目部署目标为 iOS 26，直接使用 SwiftUI 原生 `GlassButtonStyle`、`glassEffect` 和 `GlassEffectContainer`。
- 普通房间按钮使用交互式 regular glass；主操作使用 prominent glass；台板和语音播报使用同一形状、间距和 glass 容器。
- 所有房间按钮必须具备原生按压形变和高光反馈，并调用现有 `HapticService`：普通操作使用轻触反馈，开关使用选择反馈，结算等重要确认使用中等反馈。
- 禁用状态不触发触觉反馈；开启“减弱动态效果”时保留系统可访问性行为，不额外叠加自定义动画。

## 系统架构

### 权威状态

- Cloudflare Durable Object 是单个房间的串行协调者，负责鉴权、权限校验、原子变更和 WebSocket 广播。
- D1 持久化房间、成员、玩家和计分事件。Durable Object 不把内存状态当作唯一数据源。
- iOS 客户端和网页加入页都通过 Worker 的 HTTP API 加入，并通过同一 WebSocket 协议接收完整 `state` 快照。
- 每次成功变更后，服务端从 D1 读取完整最新快照并广播，客户端以服务端状态覆盖本地展示。

### 生产服务边界

- `cloudflare/mahjong-score-worker` 是唯一生产房间后端。
- `server/mahjong-sync-server` 中已经验证过的权限和零和转账逻辑迁移到 Durable Object，并用共享测试用例防止两套行为继续分叉。
- Netlify 网页加入页继续通过现有 HTTPS 代理访问 Worker；二维码不直接暴露 WebSocket 地址或成员令牌。

## 数据与权限规则

### 创建与加入

- `POST /rooms` 在多人模式下只创建一个绑定房主 `deviceId` 的玩家，其名称使用房主的 `displayName`。客户端传入的额外玩家数组在多人模式下不再生成虚拟玩家。
- `POST /rooms/:code/join` 仅允许加入 `active` 房间。
- 同一 `deviceId` 重复扫码是幂等恢复：返回原成员和原绑定玩家，不重复创建玩家。
- 新设备加入时创建成员和绑定玩家，随后向房间内所有 WebSocket 广播快照。
- 成员令牌必须同时匹配房间、成员和设备；被踢出后旧令牌立即失效。

### WebSocket 消息

| 消息 | 规则 |
| --- | --- |
| `give_score` | 活跃真实成员只能从自己的绑定玩家给另一个活跃玩家转分。服务端在一个事务中写入双方分数和两条事件。 |
| `table_score` | 活跃真实成员从自己的绑定玩家给台板转分；台板不存在时自动创建。 |
| `rename_player` | 成员可修改自己的名称；单人模式房主可修改未绑定虚拟玩家。 |
| `add_player` | 只用于单人模式的虚拟玩家；多人模式返回权限错误。 |
| `remove_player` | 房主可移除其他成员；普通成员只能退出自己；房主必须先转让群主或结算。 |
| `transfer_owner` | 仅当前房主可执行，目标必须是活跃、已绑定设备的真实成员。 |
| `adjust_score` | 仅单人模式房主可直接调整；多人模式必须走零和给分。 |
| `ping` | 维持连接并返回 `pong`，不改变房间状态。 |

- 所有变更都要求房间状态为 `active`。
- 输入在服务端再次校验，不能依赖 iOS 表单限制。
- Durable Object 的串行执行保证同一房间内的操作顺序；D1 原子 batch 保证双方分数和流水同时成功或同时失败。

### 结算

- 新增并使用 `POST /rooms/:code/settle`，请求体为 `{ "multiplier": number }`。
- 只有当前房主可以结算，倍率范围为 `(0, 1_000_000]`。
- 结算事务写入倍率、`ended_at`、每个玩家的 `multiplier_score` 和 `result`。
- `score > 0` 为 `win`，`score < 0` 为 `lose`，`score == 0` 为 `draw`。
- 结算成功后广播最终快照；所有计分和成员变更立即停止。
- 客户端不再先调用空请求 `/end`。旧 `/end` 路由不用于正常结算，并收紧为房主权限以兼容旧客户端。

## 客户端状态流

1. 创建或加入 HTTP 请求成功后，保存 `roomCode` 与 `memberToken`。
2. 立即建立 WebSocket；收到首个 `state` 后进入房间内容。
3. 用户操作只发送意图，不直接永久修改本地分数。
4. 操作发送后显示局部进行中状态并防止重复点击。
5. 收到新快照后更新列表、分数、流水、群主和房间状态，并解除进行中状态。
6. WebSocket 中断时保留最后快照，显示“正在重新连接”，暂停会改变房间的按钮。
7. 重连成功后由完整快照校正状态，不重放本地操作队列。

## 错误处理

- **网络中断**：显示“网络连接似乎出现问题，正在重新连接”，保留当前页面和数据，指数退避重连并提供手动重试。
- **服务不可达且没有快照**：显示可重试的连接页，不显示“你已退出房间”。
- **成员被移除或主动退出**：只有服务端明确返回成员不存在或令牌失效时，才显示“你已退出房间”。
- **房间已结束**：展示最终结算结果和历史明细，禁用房间变更按钮。
- **权限变化**：转让群主后立即按新快照刷新可见操作；旧房主的并发操作由服务端拒绝。
- **非法给分**：弹框内显示可操作的校验信息，服务端拒绝不会改变任何分数。
- **发送后断线**：不自动重发。重连后的服务端快照决定操作是否已经生效。

## iOS 组件改造

- `MahjongScorekeeperView` 保留页面编排职责，拆出房间工具栏、邀请 sheet、紧凑给分弹框和玩家行等小组件，避免继续扩大单文件复杂度。
- `MahjongScoreViewModel` 负责房间状态、权限派生、进行中操作、邀请显示、语音事件去重和用户可读错误。
- `MahjongRealtimeClient` 负责统一消息编码、连接状态、心跳和重连，不包含业务权限判断。
- `MahjongScoreService` 负责创建、加入、快照、结算和邀请 URL，不再用 `/end` 代替结算。
- 新增房间范围的 Liquid Glass 控件封装，集中处理形状、prominent 层级、禁用状态和 `HapticService`，确保所有按钮反馈一致。

## 验证策略

### Worker 与 D1

- 创建多人房间只产生房主玩家。
- 两个不同设备加入后分别只产生一个绑定玩家；重复加入不重复创建。
- `give_score` 和 `table_score` 保持总分为零并产生两条正确流水。
- 非本人给分、普通成员直接调分、非房主转让或结算均被拒绝。
- 被移除成员的旧令牌无法重新连接。
- 结算正确写入倍率分、胜负和结束时间，结算后所有写操作被拒绝。
- Durable Object 广播的快照与 HTTP 获取的快照一致。

### iOS

- ViewModel 测试覆盖初始房主、扫码新增、权限变化、网络重连、被踢出和结算状态。
- Service 测试确认 `/settle` 请求体、Bearer token、邀请 HTTPS URL 和错误映射。
- RealtimeClient 测试确认每种消息字段和断线不重发策略。
- UI 验证覆盖二维码底部 sheet、点击链接复制、紧凑给分框、Liquid Glass 控件、按压与触觉反馈、深色模式和减弱动态效果。

### 端到端

- 使用两台设备或一台设备加网页加入页完成：创建、扫码、加入、给分、台板、转让群主、退出、重连和结算。
- 验证房主反复打开二维码仍可加入；结算后同一二维码只能查看结束状态。
- 部署 Worker 后通过生产 HTTPS 代理和直连 Worker 各执行一次健康检查与创建/加入/结算冒烟测试。

## 部署顺序

1. 先部署兼容新旧客户端的 Worker 协议和 `/settle` 路由。
2. 执行 D1 迁移并运行生产冒烟测试。
3. 再发布使用新协议和新交互的 iOS 客户端。
4. 观察 Worker 错误率、WebSocket 重连和结算失败；确认稳定后再清理旧 `/end` 兼容路径。

## 完成标准

- 房主创建房间后只看到自己，并可随时拉起有效二维码邀请。
- 扫码成员实时出现在房间中，且不会出现重复玩家。
- 所有多人操作在线上 Worker 可用，不再返回 `Unsupported message type`。
- 给分与台板始终零和，倍率结算结果正确持久化。
- 普通断网不会显示“你已退出房间”，重连后状态自动恢复。
- 房间模块全部使用一致的 iOS 26 Liquid Glass 与按压/触觉反馈。
- iOS 测试、Worker 测试、类型检查和生产冒烟测试全部通过。
