# 飞书游戏元数据规范化设计

## 目标

规范“游戏汇总”表的字段展示与游戏元数据，使玩法说明、来源链接、人数范围和时间范围可以独立维护，并为后续审核与应用导入提供稳定数据。

## 操作范围

- Base：`PSxwbNUEMajAo3seqWicFAqsnUH`
- 数据表：`游戏汇总`（`tblCkPcGwjLMlxRw`）
- 调整字段顺序的视图：`所有反馈`（`vew2IwHmj4`）
- 数据更新范围：18 条有效游戏记录
- 排除范围：8 条仅含日期、没有完整游戏信息的占位记录

## 字段顺序

“所有反馈”视图按以下顺序展示字段：

1. `name`
2. `description`
3. `emotion`
4. `players_min`
5. `players_max`
6. `tool`
7. `time_min`
8. `time_max`
9. `icon_homepage`

11. `state`

13. `附件`

其他三个视图保持现状。字段数据属于同一张表，因此记录内容会自然同步到所有视图。

## 来源链接迁移

- 从新导入游戏的 `description` 中提取 `【内部来源】` 后的 URL。
- 将 URL 写入现有的 `链接` URL 字段。
- `description` 只保留整理后的玩法，不保留来源标记或 URL。
- 没有可靠来源链接的旧记录保持“链接”为空，不推测或补造来源。

## 人数评估规则

- `players_min` 和 `players_max` 表示完成该玩法所需的总参与人数。
- 主持人必须参与或持续主持时，计入总人数。
- 根据游戏机制、分组要求、对抗关系和淘汰方式分别评估，不统一套用固定范围。
- 最小人数应保证玩法逻辑成立；最大人数应保证等待时间和互动质量仍然合理。
- 数值必须为整数，并满足 `players_min <= players_max`。

## 时间评估规则

- `time_min` 和 `time_max` 的单位均为分钟。
- 时间范围表示完成一轮正常体验所需的合理时长，包括必要的说明、轮流互动和结算。
- 根据玩家轮次、是否分组、是否淘汰、思考成本和主持准备时间分别评估。
- 数值必须为整数，并满足 `time_min <= time_max`。

## 数据完整性与验证

写入前读取真实字段、视图顺序和目标记录；写入后回读并检查：

- 18 条有效记录均具备人数与时间上下限。
- 17 条本次采集记录的来源链接已迁移到“链接”。
- `description` 中不再含 `【内部来源】` 或来源 URL。
- 所有范围满足最小值不大于最大值。
- 8 条占位记录未被修改。
- “所有反馈”视图字段顺序符合本设计，其他视图未调整。

## 后续导入约束

今后从网页采集游戏时，来源 URL 直接写入“链接”，玩法只写入 `description`；每条新游戏在进入待审核状态前，必须完成时间和人数范围评估。
