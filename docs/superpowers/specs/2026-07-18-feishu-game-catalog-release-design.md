# 飞书游戏目录审核发布系统设计

日期：2026-07-18
状态：已确认，待实施计划

## 1. 目标

建立一条可扩展到成百上千个游戏的内容生产链路：用户提供网页链接，系统将网页中的游戏内容重新整理后写入飞书多维表格草稿；用户在飞书完成人工审核；用户指定发布批次后，发布工具只导出审核通过的记录，将授权图片发布到 CDN，将版本化游戏目录写入 iOS 工程，并完成校验、构建和发布报告。

本系统不做游戏目录热更新。新游戏和游戏文字内容只能随新的 App 版本交付。图片允许从 CDN 按需下载并在设备本地缓存，但 CDN 图片本身不能决定 App 中出现哪些游戏。

## 2. 已确认的产品约束

- 飞书是内容采集、编辑和审核后台，不是 App 运行时数据源。
- App 不包含飞书访问凭据，也不直接读取飞书。
- 网页内容使用统一、简洁的自有表述重新整理，保持玩法含义，不逐字搬运长篇正文。
- 来源链接仅保存在飞书内部，用于审核、查重和追溯，不在 App 中展示。
- 不搬运来源网页图片。只有用户提供、用户拥有授权、明确获准使用或另行生成的图片可以发布。
- 只有用户可以把游戏标记为“审核通过”。自动采集最多把记录推进到“待审核”。
- 发布必须由用户明确指定批次后执行，不允许定时或静默自动发布。
- 默认不删除或下线游戏；下线必须由飞书中的显式状态表达。
- 游戏数据随 App Store 版本发布；图片使用 Cloudflare R2/CDN，按需下载并缓存。

## 3. 当前系统现状

### 3.1 飞书

用户提供的入口是飞书电子表格链接，但实际内容是嵌入式多维表格，数据表名为“游戏汇总”。当前字段共 12 个：

- `state`
- `time_min`
- `emotion`
- `name`
- `players_max`
- `icon_homepage`
- `players_min`
- `time_max`
- `icon_detailpage`
- `tool`
- `附件`
- `description`

当前主要问题：

- 缺少稳定游戏 ID、玩法规则、语音脚本和发布版本信息。
- `time_min` 是数字而 `time_max` 是文本，字段类型不一致。
- 状态只有“已上线/未上线”，无法表达采集、审核、打包和正式上线阶段。
- 现有视图名称沿用了问题反馈模板，不适合作为内容审核工作台。
- 当前只有“真真假假”一条基本完整记录，其他样例记录不满足发布条件。

### 3.2 iOS App

游戏目录当前来自 `PartyGames/Resources/default_games.json`，`Game` 模型包含：

- `id`
- `name`
- `players`
- `playerMin` / `playerMax`
- `type`
- `tags`
- `rules`
- `voiceScript`
- `cardDescription`
- `badge`
- `duration`
- `detailIntro`
- `preparation`
- `startButtonLabel`

当前 `GameStore` 会优先读取 UserDefaults 中旧的完整游戏数组。这个行为会让老用户在安装新版后继续看到旧目录，因此不能直接支持版本化目录发布。当前 `AssetStore` 主要支持一个内置图片和本地缓存图片，也不具备通用 CDN 下载、版本隔离和失败回退能力。

## 4. 方案选择

### 4.1 采用：版本化构建时发布流水线

飞书负责采集和审核；发布工具读取指定批次的审核通过记录，生成版本化目录并修改 App 工程；图片发布到 R2；App Store 版本负责交付新目录。

选择原因：

- 满足人工审核和不做目录热更新的要求。
- 可在发布前执行完整校验、差异审查和构建验证。
- 发布产物可复现、可追踪、可回滚。
- 能把飞书编辑体验与 App 运行稳定性隔离开。

### 4.2 不采用：手工复制 JSON

手工操作在数据增长后容易出现重复 ID、数组格式错误、图片错配、漏发和不可追溯问题。

### 4.3 不采用：App 实时读取 CMS

该方式会形成目录热更新，并引入运行时权限、网络依赖、接口限额和线上数据突变，不符合本项目要求。

## 5. 总体架构与数据流

```text
用户提供一个或多个网页链接
  -> 网页读取与内容抽取
  -> 重新整理与标准化
  -> 来源、名称、内容指纹查重
  -> 写入飞书草稿/待审核
  -> 用户审核、修改并标记审核通过
  -> 用户指定发布批次
  -> 发布工具读取该批次审核通过记录
  -> 严格校验与差异报告
  -> 下载飞书中的授权图片并生成派生尺寸
  -> 以内容哈希上传 Cloudflare R2
  -> 生成版本化 App 游戏目录
  -> iOS 单元测试、回归测试和构建
  -> 生成发布报告、代码变更和回滚快照
  -> 随 App Store 新版本交付
```

发布工具是唯一允许把飞书审核数据转成 App 正式目录的边界。采集工具不能直接修改 App 正式目录。

## 6. 飞书数据模型

第一阶段继续使用现有“游戏汇总”表，避免过早拆表。上千条游戏仍可由单表承载。另建“发布批次”表记录每次发布的版本、数量、校验结果和代码提交标识。

### 6.1 身份与来源字段

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `game_id` | 单行文本 | 必填、全局唯一、发布后不可修改，使用稳定英文 slug |
| `name` | 单行文本 | 必填，用户可编辑 |
| `source_url` | URL | 内部追溯字段，不进入 App |
| `source_title` | 单行文本 | 来源页面标题 |
| `source_checked_at` | 日期时间 | 最近读取来源的时间 |
| `content_hash` | 单行文本 | 标准化内容的哈希，用于增量识别和查重 |

### 6.2 玩法与展示字段

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `players_min` | 数字 | 必填整数，至少为 1 |
| `players_max` | 数字 | 必填整数，不小于最少人数 |
| `time_min` | 数字 | 必填整数分钟 |
| `time_max` | 数字 | 必填整数分钟，不小于最短时间；迁移现有文本字段 |
| `emotion` | 多选 | 映射首页情绪筛选 |
| `tool` | 多选或单选 | 映射游戏类型和道具分类 |
| `tags` | 多选 | 用于推荐、筛选和标签展示 |
| `description` | 多行文本 | 卡片短描述 |
| `badge` | 单行文本 | 可选角标 |
| `detail_intro` | 多行文本 | 详情页介绍 |
| `rules` | 多行文本 | 必填，一行一条规则 |
| `preparation` | 多行文本 | 可选，一行一项准备事项 |
| `voice_script` | 多行文本 | 必填，一行一段语音文案 |
| `start_button_label` | 单行文本 | 可选，空值使用 App 默认文案 |
| `sort_priority` | 数字 | 同优先级使用稳定 ID 排序，保证构建可复现 |

### 6.3 图片与权利字段

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `icon_homepage` | 附件 | 首页卡片原始图 |
| `icon_detailpage` | 附件 | 详情页原始图，可与首页图相同 |
| `asset_rights` | 单选 | 自有、已授权、AI 生成、待确认 |
| `image_focal_x` / `image_focal_y` | 数字 | 可选，0 到 1，用于不同尺寸裁切的视觉焦点 |
| `image_home_url` | URL | 发布工具写回的 CDN 地址 |
| `image_detail_url` | URL | 发布工具写回的 CDN 地址 |

`asset_rights=待确认` 或图片缺失时禁止发布。来源网页图片不得作为附件进入发布链路。

### 6.4 审核与发布字段

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `state` | 单选 | 草稿、待审核、审核驳回、审核通过、已打包、已上线、已下线 |
| `review_comment` | 多行文本 | 驳回原因或审核备注 |
| `reviewed_by` | 人员 | 审核人 |
| `reviewed_at` | 日期时间 | 审核时间 |
| `release_batch` | 单行文本 | 例如 `2026-08-01` |
| `catalog_version` | 单行文本 | 写入正式目录的版本号 |
| `minimum_app_version` | 单行文本 | 该游戏要求的最低 App 版本 |
| `packed_at` | 日期时间 | 构建产物生成时间 |
| `released_app_version` | 单行文本 | 正式上线的 App 版本 |

状态机：

```text
草稿 -> 待审核 -> 审核驳回 -> 待审核
              -> 审核通过 -> 已打包 -> 已上线 -> 已下线
```

采集流程只能写入“草稿”或“待审核”。“审核通过”必须由用户在飞书中设置。“已打包”在发布工具成功生成并验证产物后写入；“已上线”只能在用户确认对应 App 版本已经上线后写入。

### 6.5 视图

- 采集草稿
- 待用户审核
- 审核驳回
- 审核通过
- 本批待发布
- 已打包
- 已上线
- 数据异常

视图只改善协作，不作为发布权限边界。发布工具仍按字段值执行服务端过滤和校验。

## 7. 网页采集流程

### 7.1 输入

用户可提供一个或多个网页 URL。单个页面可能产生一条或多条游戏记录。

### 7.2 提取与整理

- 读取游戏名称、适合人数、时长、道具、玩法步骤、准备事项和安全提醒。
- 保持玩法含义，使用项目统一语气重新整理，不复制长篇原文。
- 未能从来源确认的事实保持为空并列入审核问题，不自行虚构。
- 不读取或写入来源网页图片。
- 把来源 URL 和读取时间写入内部字段。

### 7.3 查重

写入前依次检查：

1. 规范化后的来源 URL；
2. 稳定 `game_id`；
3. 规范化游戏名称；
4. 玩法内容哈希。

发现疑似重复时不覆盖已有记录，而是生成冲突报告，请用户决定合并、更新还是保留两个版本。

### 7.4 批量与失败恢复

- 大批网址按可验证批次处理，不一次把全部内容留在内存中。
- Base 批量读写每批不超过平台限制，连续写入同一表时串行执行。
- 每批写入后回读确认记录数、稳定 ID、状态和关键字段。
- 单条失败不影响已成功写入的记录；失败列表包含 URL、阶段和原因。

## 8. 发布流程

### 8.1 发布触发

用户明确指定发布批次，例如：“发布 `2026-08-01` 批次中所有审核通过的游戏”。没有批次和明确发布指令时不得修改 App 正式目录或上传 CDN。

### 8.2 读取与快照

- 只读取 `state=审核通过` 且 `release_batch` 等于目标批次的记录。
- 读取全部分页，记录预期数量并断言实际处理数量一致。
- 在修改工程或上传图片前保存只读输入快照和校验报告。

### 8.3 发布前硬校验

以下任一问题都会阻止整个批次发布：

- `game_id` 缺失、非法、重复或与既有游戏冲突；
- 必填内容为空；
- 最少人数大于最多人数；
- 最短时间大于最长时间；
- 数字字段混入文本；
- 情绪、标签、道具类型无法映射到 App 枚举；
- 规则或语音脚本解析后为空；
- 图片缺失、损坏或素材权利未确认；
- 相同游戏在同一批次被同时标记为发布和下线；
- 生成目录不能由当前 Swift 模型解码。

### 8.4 差异预览

发布工具在产生外部写入前生成：

- 新增游戏列表；
- 修改游戏列表和字段级差异；
- 下线游戏列表；
- 未变化游戏数量；
- 图片新增、复用和替换数量；
- 校验警告和阻塞错误。

下线和 ID 变化必须单独突出显示。默认不把飞书中缺失的既有游戏解释为删除。

### 8.5 图片发布

- 从飞书附件下载用户提供的原图。
- 校验文件类型、像素尺寸、方向和文件大小。
- 根据首页和详情页需求生成派生图片，保留视觉焦点。
- 使用内容 SHA-256 生成不可变对象名，例如 `games/<game_id>/<hash>-home.jpg`。
- 相同哈希复用已有对象，避免重复上传。
- R2 写入密钥只存在于本地安全环境或 CI Secret，不进入仓库或 App。
- CDN 对图片开放只读访问；写入接口不对 App 暴露。

### 8.6 目录生成

发布工具生成版本化目录，包含：

- `schemaVersion`
- `catalogVersion`
- `generatedAt`
- 游戏记录数组
- CDN 图片 URL

数组顺序由 `sort_priority` 和 `game_id` 决定，确保同一输入产生字节稳定的结果。目录生成器不得写入来源 URL、审核备注、人员信息或飞书资源标识。

### 8.7 工程修改与验证

- 更新 App 内置目录资源。
- 运行目录 schema、映射、重复 ID 和迁移测试。
- 运行 Swift 回归测试和 iOS Simulator 构建。
- 验证离线启动、图片占位、缓存命中、失败重试和收藏/历史兼容。
- 生成发布报告、代码差异和回滚快照。
- 只有全部验证通过，才把记录推进到“已打包”。

## 9. iOS 目录与迁移设计

### 9.1 正式目录来源

内置版本化目录是正式游戏集合的唯一来源。UserDefaults 不再保存一份可覆盖内置目录的完整游戏数组。

### 9.2 用户数据保留

- 收藏和历史记录继续只保存稳定 `game_id`。
- 新版本加载目录后，过滤已经不存在或明确下线的引用。
- 新增游戏自动进入新目录，无需覆盖用户个人数据。
- 若未来保留本地管理者编辑，只能保存独立 override，并明确合并规则；第一阶段不把本地编辑纳入正式发布链路。

### 9.3 Schema 版本

- `schemaVersion` 表示 JSON 结构版本。
- `catalogVersion` 表示内容发布版本。
- App 对未知的更高 schema 明确失败并回退到随包的最后可解码目录，不静默加载部分数据。

### 9.4 类型映射

当前 App 的 `GameType` 只支持无道具、喝酒和骰子。实施时需要明确扩展轻道具和扑克等类型，或将“道具分类”和“运行组件类型”拆成两个字段，避免把展示分类强行绑定到实现能力。

## 10. 图片客户端设计

- 游戏模型保存首页图和详情图的 CDN URL。
- 图片先查内存缓存，再查磁盘缓存，最后访问 CDN。
- 磁盘文件以 URL 中的内容哈希作为缓存键，天然实现版本隔离。
- 下载到临时文件后原子移动，避免中断产生损坏缓存。
- 解码时下采样，避免上千张大图造成内存峰值。
- 首页只预取当前卡片附近的小窗口；游戏库按可见区域懒加载。
- 网络失败显示稳定占位图，并提供有限次数、带退避的重试。
- 离线时继续使用已有磁盘缓存；没有缓存时不阻塞游戏文字内容。

## 11. 发布记录与回滚

“发布批次”表至少记录：

- 批次 ID；
- 目录版本；
- 目标 App 版本；
- 输入记录数；
- 新增、修改、下线数量；
- 图片上传和复用数量；
- 校验结果；
- 输入快照哈希；
- 生成目录哈希；
- Git 分支/提交或 PR 标识；
- 打包时间和最终上线时间。

回滚不依赖实时修改飞书。回滚通过恢复上一版目录和对应代码、重新构建 App 完成。R2 使用不可变哈希对象，因此旧版本图片仍可访问。对象清理另设保留策略，不与发布动作绑定。

## 12. 安全、权限与密钥

- 采集阶段需要 Base 记录写入权限；字段和视图改造需要单独的字段/视图写入权限。
- 发布阶段需要 Base 只读权限、飞书附件下载权限和 R2 写入权限。
- 飞书权限按最小范围申请；读取和写入授权分开。
- App 不包含飞书 token、R2 Secret 或发布工具配置。
- 日志不得输出访问令牌、附件临时下载凭据或密钥。
- 外部上传、字段结构修改和正式发布都必须有用户明确指令。

## 13. 测试策略

### 13.1 采集工具

- 单页单游戏、单页多游戏和字段缺失样例；
- 来源 URL、名称和内容哈希查重；
- 不写入来源图片；
- 批次部分失败和回读验证；
- 状态永远不会由采集工具设置为审核通过。

### 13.2 发布工具

- Base 分页完整性和条数断言；
- 必填字段、范围、枚举和 ID 校验；
- 多行文本到数组的稳定转换；
- 相同输入产生稳定目录；
- 图片哈希去重和上传失败回滚；
- 差异分类和默认不删除；
- 阻塞错误时不修改正式目录、不推进状态。

### 13.3 iOS App

- 新旧 catalog schema 解码；
- 老用户从旧 UserDefaults 游戏数组迁移；
- 收藏和历史记录保留；
- 新版本目录覆盖旧目录而不覆盖个人数据；
- CDN 图片缓存、离线回退、损坏缓存恢复和下采样；
- 首页、游戏库、详情页在大目录下的加载性能；
- 全量回归和 Simulator 构建。

## 14. 实施阶段划分

### 阶段一：飞书内容工作台

- 迁移字段类型；
- 新增采集、审核和发布字段；
- 建立审核视图和状态规则；
- 保留现有记录并输出迁移报告。

### 阶段二：网页采集工具

- URL 读取、内容抽取、重新整理和查重；
- 写入草稿/待审核；
- 批量处理、失败报告和回读验证。

### 阶段三：发布工具与 CDN

- 审核通过记录导出；
- 强校验、差异预览和版本化目录生成；
- 飞书附件下载、图片派生、R2 哈希上传；
- 发布批次记录和回滚快照。

### 阶段四：iOS 目录与图片加载

- 版本化 catalog 模型；
- 修复旧 UserDefaults 覆盖新版目录的问题；
- 保留收藏和历史；
- CDN 图片加载、磁盘缓存、离线回退和性能优化。

### 阶段五：端到端试发布

- 使用少量审核通过记录跑完整链路；
- 对比飞书、生成目录和 App 页面；
- 验证构建、缓存、升级和回滚；
- 形成以后每批发布使用的固定检查清单。

## 15. 验收标准

- 用户提供网页后，可以得到可追溯、无来源图片、状态为草稿或待审核的飞书记录。
- 用户能在飞书中完成审核、驳回和批次选择。
- 未审核、字段不完整或素材权利未确认的记录无法进入正式目录。
- 同一输入可以稳定生成相同目录和图片对象名。
- 新游戏只随 App 新版本出现，不因飞书修改而在运行时变化。
- 图片按需从 CDN 下载并缓存，网络失败不阻塞游戏文字内容。
- 老用户升级后能看到新版目录，同时保留有效收藏和历史。
- 每次发布都有输入快照、差异报告、校验报告、目录版本和代码标识。
- 任一发布步骤失败时，上一版 App 目录和 CDN 图片仍保持可用。
