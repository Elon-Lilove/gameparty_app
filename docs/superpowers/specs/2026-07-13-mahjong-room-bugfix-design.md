# 麻将计分器房间交互与网页一致性修复设计

## 背景

当前麻将计分器已具备 iOS 房主端、扫码网页玩家端、Cloudflare Worker 房间状态与 WebSocket 同步能力，但实际使用中出现五类问题：全局按钮选项产生非预期阴影；网页房间与房主端界面和功能不一致；房间给分没有可靠反馈或未生效；结算操作无反馈；通知推荐页重复点击没有响应。

本次修复同时对麻将计分器内所有按钮进行有效性审计，确保每个操作具备明确的可用条件、即时反馈、服务端结果和失败恢复行为。

## 目标

- 普通选项按钮不再出现全局玻璃阴影，仅明确使用房间 Liquid Glass 控件的位置保留原生玻璃效果。
- 扫码网页在布局、功能和房间状态上与 iOS 房主端对齐。
- iOS 与网页使用同一 Worker 协议和服务端快照，玩家与分数实时一致。
- 给分、台板、改名、退出、转让房主和结算均有发送中、成功和失败反馈。
- 所有会改变房间状态的按钮防止重复提交，并在断线、权限不足或服务端拒绝时恢复可操作状态。
- 通知推荐页的主按钮始终有可感知响应；系统权限已拒绝时引导用户前往设置。
- 建立麻将计分器全量交互审计和自动化回归测试。

## 非目标

- 不把 SwiftUI 与网页重写为跨平台 UI 框架。
- 不改变现有房间权限模型：房主专属操作不会开放给普通扫码玩家。
- 不增加聊天、好友、排行榜或与计分无关的新功能。
- 不在本次修复中重做应用其他游戏页面。

## 方案选择

采用“同协议、双端同构界面”：保留 SwiftUI iOS 端和 TypeScript 网页端，两端使用相同的信息层级、功能入口、交互状态与 Worker 协议。相比远程映射房主界面，此方案不依赖房主设备持续在线；相比跨平台重写，改动范围和发布风险更可控。

## 组件与界面

### 全局按钮样式

- `HapticPlainButtonStyle` 只提供按压缩放、透明度和触觉反馈，不再附加 `glassEffect`。
- `RoomGlassButton` 和 `RoomGlassToggle` 是房间内显式玻璃效果的唯一入口。
- 禁用按钮不触发操作或触觉反馈，视觉上使用统一的禁用透明度。

### iOS 房间

- 保留房间工具栏、玩家列表、给分详情和底部结算区。
- 给分、台板、改名、移除、退出、转让房主和结算分别维护进行中状态。
- 操作发送后立即禁用对应按钮并显示进度；收到服务端快照后解除状态。
- 收到服务端错误、发送失败或连接中断时解除状态并显示用户可理解的错误。
- 多人房间不永久乐观修改分数，以服务端快照作为唯一展示结果。

### 网页玩家房间

- 页面结构对齐 iOS：房间标题和连接状态、横向工具栏、玩家表格、给分详情、结算状态和底部操作区。
- 普通玩家可使用改名、给其他玩家分、台板、语音偏好、给分详情和退出房间。
- 添加玩家、转让房主和结算仅在当前网页成员成为房主时显示或启用。
- 网页给分使用页面内弹框，校验 `1...1_000_000` 的整数，不使用浏览器 `prompt`。
- 结算结果展示倍率分和胜负状态；进行中展示实时原始分数。
- 小屏设备保持触控目标不小于 44 点，工具栏可横向滚动，玩家行与房主端保持相同信息顺序。

## 实时状态与操作反馈

### 连接状态

- WebSocket 状态分为连接中、已连接和重连中。
- 会改变房间状态的按钮只有在房间为 `active`、成员有权限且 WebSocket 已连接时可用。
- 断线时保留最后快照并显示重连提示，不把普通网络故障显示为退出房间。

### 操作生命周期

1. 用户点击后执行本地输入与权限校验。
2. 创建带操作类型的 pending 状态并禁用重复提交。
3. 通过 WebSocket 发送意图，不永久修改多人房间分数。
4. 收到包含预期变化的新快照后清除 pending，并显示短暂成功反馈。
5. 收到错误或连接失败时清除 pending，显示错误并允许重试。
6. 超时仍未收到快照时解除 pending，提示刷新或重试，不自动重发计分操作。

### 给分

- 来源固定为当前设备绑定的活跃玩家，目标为另一名活跃玩家。
- 服务端原子地从来源扣分、给目标加分，并写入两条流水。
- iOS 和网页均禁止空值、零、负数、小数和超过上限的数值。
- 分数和流水在两端收到相同服务端快照后更新。

### 结算

- 房主输入有效倍率后调用 `POST /rooms/:code/settle`，不调用旧 `/end` 代替结算。
- 提交时按钮显示“结算中”且不可重复点击。
- 成功后关闭输入框、展示结算详情并禁用所有房间变更按钮。
- 失败时保留房间进行中状态，关闭进度并展示具体错误。

## 通知推荐页

- 使用与设计图按钮位置一致的真实可访问按钮热区，并提供按压反馈。
- 请求期间禁用重复点击并显示请求中反馈。
- `notDetermined` 状态调用系统授权请求。
- `authorized`、`provisional` 或 `ephemeral` 状态直接完成引导。
- `denied` 状态不重复调用无效的系统请求，显示说明并提供前往系统设置入口。
- 请求异常时保留页面并显示错误，用户可以再次尝试。
- “或许以后”继续记录延后时间并立即退出推荐页。

## 全量按钮审计

为麻将计分器建立交互矩阵，逐项检查：开始多人、开始单人、最近房间打开/结束/删除、添加玩家、邀请链接复制、转让房主、语音开关、台板开关、台板给分、改名、给分、移除玩家、退出房间、给分详情、结算和弹框确认/取消。

每个按钮必须验证：

- 显示条件和权限正确。
- 可点击与禁用条件正确。
- 点击后立即产生视觉或触觉反馈。
- 重复点击不会生成重复操作。
- 成功后界面与服务端状态一致。
- 失败、断线和超时时有可见反馈并可恢复。

## 测试策略

### Worker

- 保留并扩展创建、加入、给分、台板、权限、结算和结算后拒绝写操作测试。
- 验证每次成功操作广播的新快照与 HTTP 快照一致。

### iOS

- 为操作 pending 状态、错误清理、给分快照更新和结算反馈增加 ViewModel 测试。
- 为通知权限状态决策抽出可测试逻辑，覆盖未决定、已允许、已拒绝和请求失败。
- 通过 iOS Simulator 构建与测试验证 SwiftUI 和 UIKit 依赖。

### 网页

- 抽离房间状态和操作校验逻辑，覆盖权限、连接状态、重复提交、数值校验和结算状态。
- 运行类型检查与生产构建。
- 使用浏览器验证窄屏布局、所有按钮、弹框、重连提示和实时快照刷新。

### 端到端

- 以 iOS 房主加网页玩家完成创建、扫码加入、双向给分、台板、改名、给分详情、转让房主、断线重连和结算。
- 确认任一端操作后两端玩家、分数、流水、房主和结算状态一致。
- 对部署环境执行只产生临时房间的冒烟测试，并记录部署版本，避免本地通过但线上仍运行旧代码。

## 发布与风险控制

1. 先确认并部署兼容两端的 Worker。
2. 再发布网页，使扫码玩家使用新界面和新操作状态。
3. 最后发布 iOS 客户端。
4. 每一步执行健康检查和跨端冒烟测试；若线上版本未更新，不把本地测试通过视为问题已修复。

## 完成标准

- 普通按钮选项没有非预期阴影，房间玻璃控件范围明确。
- 网页与房主端的结构、功能入口和结算展示一致，权限差异符合角色规则。
- iOS 与网页之间玩家、分数、流水和房间状态实时一致。
- 给分与结算能够生效，且成功、失败、断线和重复点击都有明确反馈。
- 通知推荐页主按钮单次点击即可产生可见结果，已拒绝权限时可前往设置。
- 麻将计分器交互矩阵全部通过，Worker 测试、网页测试/构建和 iOS 模拟器测试/构建无新增失败。
